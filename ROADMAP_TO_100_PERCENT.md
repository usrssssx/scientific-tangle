# План доведения системы до 100% готовности

## 0. Что считать 100% готовностью

Система готова к промышленной эксплуатации, когда она:

- принимает реальные архивы документов и экспериментальных данных без ручной подготовки;
- корректно извлекает сущности, связи, таблицы, числовые условия и единицы измерения;
- хранит данные в версионируемом графе знаний;
- поддерживает сложные запросы вида `материал + процесс + условия + география + период + числовые диапазоны`;
- отвечает за 3-5 секунд на типовые аналитические запросы при объёме до 1 млн сущностей;
- показывает источники, даты, уровень достоверности и историю изменений;
- выявляет противоречия и пробелы;
- разграничивает доступ к чувствительным данным;
- даёт удобный интерфейс исследователям, аналитикам, руководителям и администраторам;
- имеет мониторинг, аудит, резервное копирование, регламенты эксплуатации и оценку качества.

## 1. Этап обследования данных

### Цель

Понять, что лежит в архиве 4 ГБ, какие форматы, качество, языки, сканы, таблицы, дубликаты, конфиденциальность.

### Действия

1. Инвентаризировать архив: форматы, размеры, вложенность, кодировки, доля сканов.
2. Разделить источники по типам: статьи, патенты, отчёты, протоколы, презентации, таблицы, нормативка.
3. Определить классы доступа: public/internal/confidential/secret.
4. Составить карту доменов: гидрометаллургия, пирометаллургия, экология, переработка отходов, электрометаллургия.
5. Выделить 50-100 эталонных документов для ручной разметки.
6. Определить метрики качества: NER F1, relation extraction F1, numeric extraction accuracy, retrieval recall@k, answer faithfulness.

### Результат

- дата-каталог;
- профиль качества данных;
- набор эталонной разметки;
- правила классификации доступа;
- список критичных форматов и проблем.

## 2. Production ingestion pipeline

### Что нужно сделать

1. Перенести хранение сырых файлов в объектное хранилище: S3/MinIO/внутренний storage.
2. Добавить очередь задач: Celery/RQ/Kafka/NATS.
3. Сделать workers для парсинга PDF/DOCX/PPTX/XLSX/HTML/CSV/архивов.
4. Добавить OCR для сканов: PaddleOCR/Tesseract/корпоративный OCR.
5. Добавить table extraction: Camelot/Tabula/pdfplumber/LLM-assisted extraction для сложных таблиц.
6. Делать checksum и дедупликацию.
7. Хранить версию исходного файла, парсинга, модели извлечения и словарей.
8. Поддержать incremental reindex: пересчитывать только изменённые документы.

### Acceptance criteria

- архив 4 ГБ обрабатывается пакетно;
- ошибки парсинга не останавливают весь импорт;
- каждый документ имеет статус: queued/processing/parsed/failed/indexed/verified;
- для каждого факта можно восстановить исходный файл, страницу, фрагмент и версию extractor.

## 3. Онтология и модель данных

### Что нужно сделать

1. Зафиксировать базовые классы: Material, Process, Equipment, Property, Experiment, Publication, Expert, Facility, Finding, Technology, EconomicIndicator, Geography.
2. Зафиксировать отношения: uses_material, operates_at_condition, produces_output, described_in, validated_by, contradicts, has_numeric_constraint, expert_in, performed_by.
3. Описать доменные свойства: концентрация, температура, pH, Eh, скорость, расход, производительность, извлечение, CAPEX, OPEX.
4. Ввести единый справочник единиц и конвертаций.
5. Ввести SHACL-валидацию ограничений: например, у `NumericFact` должны быть property, unit, value/range, source, confidence.
6. Поддержать RDF/OWL/JSON-LD экспорт для совместимости.
7. Добавить entity resolution: `электроэкстракция`, `electrowinning`, `electroextraction` должны вести к одной сущности.

### Acceptance criteria

- новые домены подключаются через расширение онтологии, а не переписывание кода;
- невалидные факты не попадают в verified-слой;
- граф можно экспортировать в JSON-LD/RDF.

## 4. Хранилища production-уровня

### Рекомендуемая целевая схема

```text
Object Storage       raw files, parsed text, page images
PostgreSQL           metadata, users, tasks, audit, permissions
Neo4j / Neptune      property graph and Cypher/Gremlin traversal
Elasticsearch/Vespa  hybrid search: BM25 + vectors + filters
Redis                cache for hot queries and sessions
MLflow/registry      versions of extraction models and prompts
```

### Почему не один storage

- графовая БД нужна для обхода связей 3-4 уровня;
- поисковый движок нужен для быстрых текстовых и векторных запросов;
- PostgreSQL удобнее для транзакционных данных, аудита и прав;
- объектное хранилище безопаснее для больших исходных файлов.

### Acceptance criteria

- 1 млн сущностей и десятки миллионов связей не ломают графовый слой;
- фильтрация по числовым условиям работает индексно;
- поиск поддерживает русский/английский, BM25, embeddings, rerank;
- горячие запросы отвечают стабильно в пределах SLA.

## 5. NLP и извлечение знаний

### Слои извлечения

1. Dictionary layer — быстрые синонимы и термины.
2. NER layer — модели для материалов, процессов, оборудования, параметров.
3. Numeric layer — строгий parser чисел, диапазонов, единиц и конвертаций.
4. Relation extraction — связи между сущностями.
5. Table extraction — отдельный pipeline для таблиц экспериментов.
6. Claim extraction — выделение выводов, рекомендаций, ограничений, противоречий.
7. Human validation — экспертная корректировка и подтверждение.

### Модели

- ru/en domain NER: ruBERT/DeBERTa/XLM-R, дообученные на разметке;
- relation extraction: supervised classifier + LLM-assisted extraction with schema validation;
- embeddings: multilingual domain embeddings, отдельно оценить BGE-M3, E5, Qwen/Snowflake/Nomic под русский и английский корпус;
- reranker: multilingual cross-encoder/reranker для точности top-k;
- LLM для синтеза ответа только после retrieval и с обязательными ссылками на факты.

### Числа и единицы

Критично сделать отдельный deterministic validator:

- `мг/л`, `мг/дм³`, `г/л`;
- `м/с`, `м3/ч`, `т/сут`;
- `%`, `°C`, `руб/м3`, `USD/m3`;
- диапазоны, неравенства, интервалы применимости;
- нормализация `дм³ = л`, `мг/дм³ = мг/л`;
- контроль невозможных значений.

### Acceptance criteria

- numeric extraction accuracy близко к 100% на эталонном наборе;
- модель не путает год публикации с температурой или концентрацией;
- каждая extracted relation имеет evidence span;
- эксперт может исправить факт, и исправление становится новой версией.

## 6. Верификация, достоверность и версионирование

### Модель confidence

Факт получает confidence из нескольких факторов:

- тип источника: peer-reviewed article, patent, internal protocol, presentation, note;
- качество extraction;
- наличие эксперимента;
- число независимых подтверждений;
- свежесть;
- ручная верификация экспертом;
- противоречия с другими фактами.

### Состояния фактов

- extracted;
- normalized;
- candidate;
- verified;
- rejected;
- superseded;
- contradicted.

### Версионирование

Нужно хранить:

- кто внёс факт;
- когда;
- из какого источника;
- какой extractor/model/prompt;
- какая версия словаря;
- чем факт был заменён;
- кто подтвердил или отклонил.

### Acceptance criteria

- по любому ответу можно открыть trace: source → chunk/page → extracted fact → verification status;
- противоречивые выводы не скрываются, а группируются;
- руководитель видит темы с низкой достоверностью.

## 7. Query engine и синтез ответов

### Pipeline production-запроса

```text
user query
  -> language detection
  -> query understanding: entities, filters, ranges, intent
  -> hybrid retrieval: BM25 + vectors + metadata filters
  -> graph expansion: 1-4 hops
  -> numeric filtering and unit normalization
  -> reranking
  -> evidence pack
  -> answer generation with citations
  -> verifier/checker
  -> final structured answer
```

### Типы ответов

1. Краткая справка.
2. Литературный обзор.
3. Сравнительная таблица технологий.
4. Список экспериментов.
5. Карта связей материал → процесс → оборудование → результат.
6. Пробелы знаний.
7. Эксперты и команды.
8. Экспорт PDF/Markdown/JSON-LD.

### Acceptance criteria

- сложный запрос с 3-4 hop graph expansion отвечает за 3-5 секунд на целевом объёме;
- ответ содержит источники, даты, confidence и ограничения применимости;
- LLM не добавляет утверждения без evidence.

## 9. Безопасность и ИБ

### Production-требования

1. SSO: AD/LDAP/OIDC.
2. RBAC + ABAC: роль, подразделение, проект, уровень допуска, тип источника.
3. Классификация данных: public/internal/confidential/secret.
4. Шифрование at rest и in transit.
5. Полный аудит: поиск, просмотр, экспорт, загрузка, изменение фактов.
6. DLP-ограничения на экспорт.
7. SIEM-интеграция.
8. Secrets management.
9. Регламент удаления и ретенции данных.
10. Разделение контуров: dev/test/prod.

### Acceptance criteria

- внешний партнёр не видит внутренние отчёты;
- экспорт чувствительных данных логируется и контролируется;
- администратор видит полный audit trail;
- система проходит внутренний security review.

## 10. Производительность

### Целевые SLA

- простой поиск: <1 сек;
- сложный аналитический запрос: 3-5 сек;
- построение графа до 4 уровней: 3-5 сек;
- пакетный ingest: зависит от OCR, но должен быть возобновляемым;
- индексация новых документов: near-real-time или batch по расписанию.

### Меры

- индексирование metadata и numeric fields;
- precomputed graph neighborhoods для частых сущностей;
- query cache;
- rerank только top-N кандидатов;
- асинхронная генерация тяжёлых обзоров;
- read replicas для search/graph;
- нагрузочное тестирование на синтетическом графе 1 млн+ сущностей.

## 13. Поэтапный план внедрения

### Фаза 1 — 2-4 недели: MVP на реальном подкорпусе

- загрузить 1-5% архива;
- расширить словари под реальные термины;
- разметить 50 документов;
- проверить качество extraction;
- собрать первые отзывы исследователей.

### Фаза 2 — 1-2 месяца: инженерный прототип

- добавить object storage;
- добавить async ingest;
- подключить production search;
- подключить Neo4j/Neptune;
- сделать базовую expert curation;
- сделать интеграцию с SSO в тестовом контуре.

### Фаза 3 — 2-3 месяца: качество знаний

- дообучить NER/RE;
- внедрить numeric validator;
- добавить table extraction;
- ввести SHACL-валидацию;
- настроить confidence и contradiction detection;
- провести evaluation на эталонном наборе.

### Фаза 4 — 2 месяца: production hardening

- нагрузочные тесты;
- мониторинг;
- backup/restore;
- SIEM/audit;
- DLP/export policies;
- документация и обучение пользователей;
- пилот на 1-2 подразделениях.

### Фаза 5 — масштабирование

- подключение новых доменов;
- подключение датчиков/экспериментов;
- расширение уведомлений;
- руководительские KPI-дашборды;
- регулярный цикл переоценки качества.

## 14. Главные риски

1. Низкое качество PDF/сканов — решается OCR и ручной проверкой критичных документов.
2. Терминологическая неоднозначность — решается онтологией и entity resolution.
3. Ошибки чисел и единиц — решается отдельным deterministic numeric validator.
4. Недоверие экспертов — решается traceability и human-in-the-loop.
5. Слишком медленный граф — решается индексами, precompute, кэшем, ограничением graph expansion.
6. Утечки данных — решается ABAC/RBAC, классификацией, аудитом, DLP.
7. Галлюцинации LLM — решается evidence-only generation и verifier.

## 15. Definition of Done

Система считается готовой, если:

- обработан полный архив;
- не менее 95% документов успешно распознаны и проиндексированы;
- critical numeric extraction проходит внутренний тест качества;
- есть verified-слой знаний;
- исследователи подтверждают полезность на реальных вопросах;
- все ответы содержат источники и confidence;
- права доступа протестированы;
- аудит и мониторинг включены;
- есть инструкция эксплуатации и план восстановления.
